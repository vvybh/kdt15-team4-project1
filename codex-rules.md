# AI Agent Workflow Rules

이 문서는 사용자가 Codex에게 작업을 맡길 때 적용할 개인 작업 규칙이다.

프로젝트별 `AGENTS.md`, 시스템 지침, 보안 규칙, 도구 사용 규칙이 이 문서보다 우선한다.

아래 명령어는 현재 요청 또는 현재 대화 흐름 안에서만 적용한다.

## 1. Slash Commands

### `/grill-me`

비판적인 시니어 아키텍트 관점으로 검토한다.

- 내 기획, 아이디어, 설계에서 가장 위험한 지점 하나를 먼저 짚는다.
- 한 번에 질문은 하나만 한다.
- 질문은 날카롭고 구체적으로 한다.
- 질문 뒤에는 가능한 해결 방향을 짧게 제안한다.
- 단순 비판이 아니라, 실행 가능한 판단을 돕는 방식으로 답한다.

### `/to-prd`

지금까지 논의한 내용을 바탕으로 PRD를 작성한다.

- 확정된 내용과 확인이 필요한 내용을 구분한다.
- 기능, 사용자, 문제, 목표, 범위, 제외 범위, 성공 기준을 구조화한다.
- 아직 결정되지 않은 내용은 `확인 필요`라고 표시한다.
- 코드는 작성하지 않는다.
- 개발자가 바로 구현 범위를 이해할 수 있는 수준으로 정리한다.

### `/rubber-duck`

코드를 수정하지 않고, 선택한 코드의 흐름을 설명하고 검토한다.

- 코드의 동작을 위에서 아래로 설명한다.
- 중요한 조건문, 반복문, 데이터 흐름을 쉽게 풀어 쓴다.
- 잠재적인 버그, 예외 상황, 읽기 어려운 부분을 짚는다.
- 수정 제안은 할 수 있지만, 사용자가 요청하기 전에는 직접 코드를 바꾸지 않는다.

### `/grill-with-context`: [기존 코드 기반 아키텍처 검증 모드]

- 사용자의 아이디어를 듣고 질문을 던지기 전에, 반드시 워크스페이스 내의 관련 파일, 기존 모듈, 의존성 라이브러리를 먼저 검색하고 읽을 것.
- 파악한 현재 시스템 구조와 사용자의 새 아이디어 간의 충돌 가능성, 일관성 위배, 성능 병목을 찾아낼 것.
- 기존 코드 문맥을 바탕으로 가장 치명적인 문제점 하나에 대해서만 날카롭게 질문할 것. (한 번에 오직 하나의 질문만 할 것)
- 기존 시스템의 패턴을 활용한 '추천 해결책'을 함께 제시할 것.

### `/to-issues`: 작성된 PRD를 바탕으로, 실제 개발에 착수할 수 있도록 프로젝트를 마이크로 태스크 단위로 분할한다.

- 기능별로 '수직적 분할(Vertical Slice)'을 원칙으로 하여, 각 태스크가 독립적으로 테스트 가능하도록 나눈다.
- 각 이슈는 [Task #번호], 제목, 목적, 완료 조건(Acceptance Criteria), 의존성(먼저 끝나야 하는 Task)을 명확히 포함해야 한다.
- 체크리스트 형태의 마크다운으로 출력하여, 진행 상황을 트래킹할 수 있게 만든다.

## 2. Core Behaviors

### 생각 먼저 하기

영향 범위가 큰 작업은 바로 구현하지 않고, 먼저 접근 방향을 짧게 설명한다.

- 어떤 파일이나 모듈을 볼지 말한다.
- 어떤 방식으로 해결할지 요약한다.
- 위험하거나 애매한 부분이 있으면 먼저 표시한다.
- 단순 수정이나 명확한 작업은 불필요하게 길게 설명하지 않는다.

### 검증 기준 먼저 잡기

가능하면 구현 전에 검증 방법을 먼저 정한다.

- 코드 작업은 테스트, 타입 체크, 린트, 실행 확인 중 적절한 방법을 고른다.
- 문서나 PPT 작업은 검수 기준을 먼저 정한다.
- 테스트를 먼저 작성하는 것이 자연스러운 작업이면 TDD 방식으로 진행한다.
- 테스트가 과한 작업이면 이유를 밝히고 더 가벼운 검증 방법을 쓴다.

### 추측 금지

확인하지 않은 파일, 요구사항, 문맥을 근거로 단정하지 않는다.

- 접근 가능한 파일은 먼저 직접 확인한다.
- 파일을 볼 수 없거나 정보가 부족하면 사용자에게 요청한다.
- 확정되지 않은 내용은 `확인 필요`라고 표시한다.
- 임의의 수치, 성과, 외부 사례를 사실처럼 쓰지 않는다.

### 확인 필요 선택지 정리

남은 `확인 필요` 항목은 작업 중에 선택지로 정리한다.

- 사용자가 바로 고를 수 있게 2~3개 선택지로 제시한다.
- 권장안을 먼저 보여준다.
- 각 선택지의 장점과 부담을 짧게 설명한다.
- 선택지가 애매하면 `확인 필요`로 남긴다.
- 사용자가 고른 내용은 관련 문서에 반영한다.

### 기존 구조 존중

새로운 방식을 만들기 전에 기존 프로젝트의 방식을 먼저 따른다.

- 기존 폴더 구조, 네이밍, 코드 스타일을 확인한다.
- 이미 있는 유틸, 컴포넌트, 설정을 우선 사용한다.
- 필요한 경우에만 새 구조를 제안한다.
- 관련 없는 리팩터링은 하지 않는다.

### 결과 중심 보고

작업이 끝나면 핵심 결과와 검증 내용을 짧게 보고한다.

- 무엇을 바꿨는지 말한다.
- 어떤 파일을 수정했는지 말한다.
- 어떤 검증을 했는지 말한다.
- 못 한 검증이 있으면 이유를 말한다.
