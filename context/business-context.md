# 비즈니스 상황

## 고객 또는 사용자 유형

- 중고 물품을 사려는 사용자
- 중고 물품을 팔려는 사용자
- 중고 시세를 빠르게 확인하고 싶은 사용자
- 팀 프로젝트 평가자

## 현재 문제

- 중고 물품은 같은 제품이라도 가격 차이가 크다.
- 사용자는 적정 가격을 직접 판단하기 어렵다.
- 여러 사이트를 따로 확인해야 해서 시간이 걸린다.
- 판매자는 너무 비싸게 올리거나 너무 싸게 팔 위험이 있다.
- 구매자는 시세보다 높은 가격에 살 위험이 있다.
- 경매 방식 거래를 하려면 별도의 구조가 필요하다.

## 원하는 결과

- 사용자는 경매 웹사이트에서 물품을 판매하거나 구매한다.
- 판매자는 물품 등록 중 시세 그래프를 보고 시작가를 정한다.
- 구매자는 입찰 중 시세 그래프를 보고 입찰 여부를 판단한다.
- 가격 데이터는 경매 거래를 돕는 정보로 사용한다.
- 팀은 데이터 수집, 데이터 가공, 웹 개발 결과를 하나의 서비스 흐름으로 보여준다.

## 제안 방향

중고거래 가격 데이터를 수집하고 정리한다.
정리한 데이터는 경매 웹사이트 안에서 시세 그래프로 보여준다.
시세 그래프는 별도 기능이 아니라 판매와 구매 판단을 돕는 화면 요소다.
웹사이트는 실시간 크롤링을 하지 않고 저장된 데이터를 읽는다.
사용자는 물품을 등록하고, 다른 사용자는 경매 방식으로 입찰한다.

## 서비스 핵심 기능

- 경매 물품 등록
- 입찰 참여
- 경매 마감 결과 확인
- 물품 검색
- 판매 등록 화면의 시세 그래프
- 경매 상세 화면의 시세 그래프
- 시작가와 입찰가 판단을 돕는 가격 참고 정보
- 낙찰 뒤 관리자 검수 상태 확인

## 데이터 활용 방향

- 데이터 수집: 크롤링 중심으로 진행
- 데이터 출처: 당근마켓, 중고나라, 번개장터, KREAM
- 데이터 정리: 물품명, 가격, 날짜, 상태 등 필요한 항목 추출
- 데이터 가공: 이상치 제거, 물품별 묶기, 가격 통계 계산
- 샘플 데이터 작성: CSV
- 웹사이트 조회 데이터: SQLite
- 원본 데이터 보관: JSON 선택 사용
- 데이터 표시: 저장된 데이터를 판매 등록 화면과 경매 상세 화면에서 그래프와 요약 가격으로 제공

이상치는 전체 흐름과 크게 다른 값을 뜻한다.
예를 들어 잘못 입력된 가격이나 부품만 판매하는 글이 포함될 수 있다.

## 현재 사용 중인 도구

- Python: 데이터 수집과 정리
- HTML: 웹사이트 화면 구성
- Flask: Python 기반 웹 프레임워크
- SQLite: 데이터 저장
- requests: 공개 HTML 요청
- BeautifulSoup: HTML 데이터 파싱
- Selenium: 동적 페이지 수집 보조
- Chart.js: 시세 그래프 표시
- Open API: 사용하지 않아도 됨

## 주요 위험

- 기술적으로 크롤링이 가능해도 사이트 정책상 제한될 수 있다.
- 사이트별 이용약관과 robots.txt를 최종 확인해야 한다.
- 같은 물품을 정확히 묶는 기준이 필요하다.
- 중고 물품 상태에 따라 가격 차이가 클 수 있다.
- 데이터가 부족하면 적정 가격 신뢰도가 낮아질 수 있다.
- CSV와 SQLite의 컬럼 기준이 달라지면 화면과 계산 결과가 어긋날 수 있다.
- 관리자 검수 흐름은 실제 배송사 연동 없이 화면 상태로만 표현한다.
- 크롤링 실패 시 기존 저장 데이터로 화면을 유지해야 한다.

## 확인 필요

- 사이트별 실제 접근 가능한 데이터 항목
- 크롤링 주기
- 프로젝트 제출 마감일
