# 프로젝트 브리프

## 작업 목표

사용자끼리 중고 물품을 경매 방식으로 거래할 수 있는 웹사이트를 만든다.
이 웹사이트가 프로젝트의 주 목적이다.
여러 중고거래 사이트에서 수집한 가격 데이터는 웹사이트 안에서 보조 기능으로 사용한다.
사용자는 물품을 판매하거나 구매할 때 특정 화면에서 시세 그래프를 확인할 수 있다.
시세 그래프는 시작가, 입찰가, 구매 판단을 돕는 역할을 한다.
웹사이트는 실시간 크롤링을 하지 않고, 미리 저장해 둔 데이터를 기준으로 시세 그래프를 보여준다.

크롤링은 웹사이트의 공개된 페이지에서 필요한 정보를 자동으로 모으는 작업이다.
Open API는 외부 서비스가 정해 둔 방식으로 데이터를 요청하고 받는 기능이다.

## 대상 독자

- 팀 프로젝트 평가자
- 발표를 듣는 교수님과 수강생
- 웹사이트를 사용할 중고거래 이용자
- 팀원 4명

## 핵심 메시지

중고거래 사용자는 가격 판단이 어렵다.
이 프로젝트는 경매 방식의 중고거래 웹사이트를 제공한다.
판매자는 시세 그래프를 참고해 시작가를 정할 수 있다.
구매자는 시세 그래프를 보고 입찰 여부를 판단할 수 있다.
가격을 정하는 과정은 경매 흐름 안에서 진행된다.

## 산출물

- 사용자 간 경매 거래 웹사이트
- 중고거래 데이터 수집 코드
- 수집 데이터 정리 결과
- CSV 샘플 데이터
- SQLite 조회 데이터베이스
- 판매 또는 구매 화면 안의 물품별 시세 그래프
- 시작가와 입찰 판단을 돕는 적정 가격 참고 기능
- 발표자료
- 프로젝트 설명 문서

## 발표 또는 사용 상황

팀 프로젝트 발표에서 사용한다.
발표는 문제, 데이터 수집 방법, 데이터 가공 방법, 웹사이트 기능, 팀 역할 순서로 설명한다.
웹사이트는 경매 물품 등록, 시세 그래프 확인, 입찰, 낙찰 흐름을 기준으로 시연한다.
발표자료는 PPT 10~15장 안에서 구성한다.
발표 시간은 10분이다.

## 팀 구성과 역할

- 조장: 발표자료 준비, 발표, 팀원 관리
- 팀원 1: 데이터 크롤링, 전반적인 파이썬 코딩
- 팀원 2: 데이터 크롤링, 전반적인 파이썬 코딩
- 팀원 3: 웹사이트 개발

## 기본 흐름

1. 여러 중고거래 사이트에서 물품 데이터를 수집한다.
2. 수집한 데이터에서 물품명, 가격, 날짜, 상태 등 필요한 항목을 정리한다.
3. 샘플 데이터는 CSV로 작성한다.
4. 웹사이트가 읽을 데이터는 SQLite에 저장한다.
5. 사용자는 웹사이트에서 경매 물품을 등록한다.
6. 판매자는 등록 화면에서 저장된 데이터 기반 시세 그래프를 보고 시작가를 참고한다.
7. 구매자는 입찰 화면에서 저장된 데이터 기반 시세 그래프를 보고 입찰 여부를 판단한다.
8. 사용자는 경매 방식으로 입찰하고 낙찰 결과를 확인한다.
9. 낙찰 뒤 판매자는 물품을 관리자에게 보낸다.
10. 관리자는 물품을 검수한 뒤 구매자에게 보내는 상태를 표시한다.

## 데이터 수집 범위

- Open API 사용 여부: 사용하지 않아도 됨
- 수집할 물품 카테고리: 의류, 생필품 등
- 최종 확인 사항: 사이트 이용약관, robots.txt, 개인정보 수집 여부

robots.txt는 사이트가 자동 수집 프로그램에 허용하거나 제한하는 경로를 적어 둔 파일이다.

| 사이트 | 팀 기준 수집 가능 여부 | 최종 확인 |
| --- | --- | --- |
| 당근마켓 | 가능 | 공식 약관과 robots.txt 확인 필요 |
| 중고나라 | 가능 | 공식 약관과 robots.txt 확인 필요 |
| 번개장터 | 가능 | 공식 약관과 robots.txt 확인 필요 |
| KREAM | 가능 | 공식 약관과 robots.txt 확인 필요 |

기술적으로 데이터를 가져올 수 있는 것과 정책상 허용되는 것은 다를 수 있다.
제출 전에는 각 사이트의 이용약관과 robots.txt를 다시 확인한다.

## 적정 가격 기준

동일 상품 안에서 비슷한 조건의 거래 내역을 확인한다.
비슷한 조건은 상품 상태, 사용 기간, 구성품, 브랜드, 거래 시점 등을 기준으로 판단한다.
참고 시세는 비슷한 조건 상품의 중간 가격으로 계산한다.
시세 범위는 낮은 쪽 25% 가격부터 높은 쪽 25% 가격까지로 표시한다.
자세한 기준은 `context/pricing-design.md`를 따른다.

## 시세 그래프 사용 위치

시세 그래프는 별도 서비스가 아니다.
경매 웹사이트 안에서 판매자와 구매자의 가격 판단을 돕는 기능이다.

- 판매 등록 화면: 판매자가 시작가를 정할 때 확인한다.
- 경매 상세 화면: 구매자가 현재 입찰가가 적절한지 확인한다.
- 검색 또는 물품 상세 화면: 같은 물품의 최근 가격 흐름을 확인한다.

## 경매 방식

실제 경매 방식과 유사하게 구성한다.
판매자는 시작가를 정하고, 구매자는 더 높은 가격으로 입찰한다.
마감 시간에 가장 높은 가격을 제시한 사용자가 낙찰되는 흐름을 기본으로 한다.
구체적인 시작가, 입찰 단위, 마감 시간, 자동 연장 여부는 확인 필요.

낙찰 뒤 거래는 관리자 검수 중개 방식으로 표현한다.
판매자가 관리자에게 물품을 보내고, 관리자가 물품을 검수한 뒤 구매자에게 보내는 흐름이다.
실제 결제 기능과 회원 기능은 과제 범위에서 제외한다.

## 웹사이트 기술 스택

- 중심 기술: Python, HTML
- 웹 프레임워크: Flask
- 데이터 저장: SQLite
- 샘플 데이터 작성: CSV
- 웹사이트 데이터 사용 방식: 저장된 SQLite 데이터 사용
- 크롤링 원본 보관: JSON 선택 사용
- 기본 크롤링 방식: requests + BeautifulSoup
- 동적 페이지 보조 도구: Selenium
- 그래프 표시: Chart.js
- 화면 구현: HTML 중심

## 성공 기준

- 물품별 가격 데이터를 수집하고 정리할 수 있다.
- 경매 웹사이트 안에서 저장 데이터 기반 시세 그래프를 확인할 수 있다.
- 판매 등록과 구매 입찰 과정에서 시세 그래프가 사용된다.
- 크롤링 실패 시 기존 저장 데이터로 화면을 유지할 수 있다.
- CSV 샘플 데이터를 SQLite에 넣어 웹사이트에서 사용할 수 있다.
- 대표 상품 1개당 샘플 데이터 5개를 기준으로 시연할 수 있다.
- 시작가와 입찰 판단에 쓰는 가격 기준을 설명할 수 있다.
- 사용자가 물품을 등록하고 경매에 참여하는 흐름을 시연할 수 있다.
- 낙찰 뒤 관리자 검수 중개 흐름을 설명할 수 있다.
- 발표에서 데이터 수집, 데이터 가공, 웹사이트 구현의 연결이 명확하게 보인다.
- 확인되지 않은 내용은 `확인 필요`로 표시되어 있다.

## 확인 필요

- 사이트별 크롤링 정책 최종 확인
- 경매 시작가, 입찰 단위, 마감 시간 세부 규칙
- 수집할 세부 카테고리와 대표 상품

## 확인 필요 처리 방식

남은 확인 필요 항목은 작업 중에 Codex가 선택지로 정리한다.
사용자는 제시된 선택지 중 하나를 고른다.
선택지가 부족하면 사용자가 직접 다른 의견을 말할 수 있다.

선택지 형식은 아래처럼 작성한다.

1. 권장안: 현재 프로젝트에 가장 무난한 선택
2. 간단안: 구현이 쉬운 선택
3. 확장안: 기능은 좋지만 작업량이 늘어나는 선택
