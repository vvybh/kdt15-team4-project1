# PRD

## 문서 목적

이 문서는 경매 방식 중고거래 웹사이트의 구현 범위를 정리한다.
시세 그래프는 별도 서비스가 아니다.
판매자와 구매자가 가격을 판단할 때 웹사이트 안에서 확인하는 보조 기능이다.

PRD는 제품 요구사항 문서다.
무엇을 만들지, 누구를 위한 것인지, 어떤 기능이 필요한지 정리하는 문서다.

## 프로젝트 목표

사용자끼리 중고 물품을 경매 방식으로 거래할 수 있는 웹사이트를 만든다.
판매자는 물품을 등록하고 시작가를 정한다.
구매자는 경매에 입찰한다.
판매자와 구매자는 물품 관련 시세 그래프를 보고 가격 판단에 참고한다.

## 사용자

### 판매자

판매자는 중고 물품을 경매 방식으로 판매하려는 사용자다.

판매자는 아래 행동을 한다.

- 물품을 등록한다.
- 물품명, 설명, 사진, 시작가, 마감 시간을 입력한다.
- 등록 화면에서 시세 그래프를 확인한다.
- 시세를 참고해 시작가를 정한다.
- 경매 진행 상태와 낙찰 결과를 확인한다.
- 낙찰 뒤 물품을 관리자에게 보낸다.

### 구매자

구매자는 중고 물품을 경매 방식으로 구매하려는 사용자다.

구매자는 아래 행동을 한다.

- 경매 중인 물품을 검색한다.
- 물품 상세 화면에서 현재 입찰가와 남은 시간을 확인한다.
- 시세 그래프를 보고 입찰 여부를 판단한다.
- 입찰가를 입력하고 입찰한다.
- 낙찰 여부를 확인한다.
- 관리자가 검수한 물품을 받는 흐름을 확인한다.

### 관리자

관리자는 낙찰 뒤 거래를 중간에서 관리하는 역할이다.

관리자는 아래 행동을 한다.

- 판매자에게 물품을 받는다.
- 물품 상태를 검수한다.
- 검수 결과를 표시한다.
- 검수 완료 물품을 구매자에게 보내는 상태를 표시한다.

### 프로젝트 평가자

프로젝트 평가자는 발표와 시연을 통해 프로젝트 결과를 확인하는 사람이다.

평가자는 아래 내용을 확인한다.

- 데이터 수집과 웹사이트 기능이 연결되는가?
- 경매 흐름이 이해하기 쉬운가?
- 시세 그래프가 판매와 구매 판단에 도움이 되는가?
- 관리자 검수 흐름이 낙찰 뒤 거래 방식으로 설명되는가?
- 팀원별 역할이 명확한가?

## 핵심 문제

중고거래에서는 같은 물품이라도 가격 차이가 크다.
판매자는 시작가를 정하기 어렵다.
구매자는 현재 입찰가가 적절한지 판단하기 어렵다.
여러 사이트를 따로 확인해야 해서 시간이 걸린다.

## 해결 방향

경매 웹사이트 안에서 가격 판단에 필요한 시세 정보를 함께 보여준다.
판매자는 물품 등록 중 시세 그래프를 보고 시작가를 참고한다.
구매자는 입찰 전 시세 그래프를 보고 입찰 여부를 판단한다.
경매 흐름은 등록, 입찰, 마감, 낙찰, 관리자 검수, 구매자 전달 순서로 구성한다.
회원 기능과 결제 기능은 과제 범위에서 제외한다.

## 기능

### 1. 경매 물품 등록

판매자는 경매 물품을 등록할 수 있다.

입력 항목은 아래와 같다.

- 물품명
- 카테고리
- 설명
- 사진
- 시작가
- 경매 마감 시간
- 상품 상태
- 구성품

등록 화면에는 같은 물품 또는 비슷한 물품의 시세 그래프를 보여준다.
판매자는 그래프를 보고 시작가를 조정할 수 있다.

### 2. 경매 물품 목록

사용자는 진행 중인 경매 물품을 볼 수 있다.

목록에서 보여줄 정보는 아래와 같다.

- 물품명
- 대표 이미지
- 현재 입찰가
- 남은 시간
- 입찰 수
- 카테고리

### 3. 경매 상세 화면

사용자는 물품 상세 정보를 확인하고 입찰할 수 있다.

상세 화면에서 보여줄 정보는 아래와 같다.

- 물품명
- 사진
- 설명
- 상품 상태
- 시작가
- 현재 입찰가
- 남은 시간
- 입찰 버튼
- 시세 그래프
- 최근 비슷한 거래 가격 요약

시세 그래프는 구매자가 현재 입찰가가 적절한지 판단하는 데 사용한다.

### 4. 입찰

구매자는 현재 입찰가보다 높은 금액으로 입찰한다.
입찰 단위와 자동 연장 여부는 확인 필요다.

입찰 시 필요한 기본 검사는 아래와 같다.

- 현재 입찰가보다 높은 금액인지 확인한다.
- 최소 입찰 단위를 지켰는지 확인한다.
- 경매 마감 시간이 지나지 않았는지 확인한다.
- 판매자명과 입찰자명이 같은 경우 안내한다.

### 5. 경매 마감과 낙찰

경매 마감 시간이 지나면 가장 높은 금액을 제시한 사용자가 낙찰된다.
낙찰 결과는 판매자와 구매자가 확인할 수 있어야 한다.

낙찰 뒤 거래는 관리자가 중간에서 검수하는 방식으로 진행한다.
이 흐름은 KREAM과 유사한 검수 중개 방식이다.

검수 중개 방식은 판매자가 구매자에게 바로 보내지 않는 방식이다.
판매자가 관리자에게 물건을 보내고, 관리자가 상태를 확인한 뒤 구매자에게 보낸다.

### 6. 관리자 검수와 발송

관리자는 낙찰된 물품을 받은 뒤 상태를 확인한다.
검수 결과는 통과, 보류, 실패 중 하나로 표시한다.
검수를 통과한 물품은 구매자에게 보내는 상태로 바꾼다.

표시할 상태는 아래와 같다.

- 판매자 발송 대기
- 관리자 수령 완료
- 검수 중
- 검수 통과
- 검수 보류
- 검수 실패
- 구매자 발송 완료

### 7. 물품 검색

사용자는 물품명 또는 카테고리로 경매 물품을 검색할 수 있다.
검색 결과에서는 진행 중인 경매를 우선 보여준다.

검색 기준은 아래와 같다.

- 물품명
- 카테고리
- 가격 범위
- 경매 상태

### 8. 시세 그래프

시세 그래프는 경매 웹사이트 안의 보조 기능이다.
판매 등록 화면과 경매 상세 화면에서 보여준다.

그래프에 표시할 내용은 아래와 같다.

- 같은 물품 또는 비슷한 물품의 가격
- 거래 또는 게시 날짜
- 평균 가격
- 중간 가격
- 최저가와 최고가
- 데이터 출처

중간 가격은 가격을 낮은 순서로 정렬했을 때 가운데에 있는 값이다.
가격이 크게 튀는 데이터의 영향을 평균보다 덜 받는다.

### 9. 가격 참고 정보

웹사이트는 적정 가격을 단정하지 않는다.
대신 시작가와 입찰가 판단에 참고할 가격을 보여준다.
가격 참고 정보는 실시간 크롤링이 아니라 저장된 데이터를 기준으로 만든다.

표시 예시는 아래와 같다.

- 비슷한 물품의 중간 가격
- 최근 거래 가격 범위
- 현재 입찰가가 시세보다 높은지 낮은지에 대한 간단한 안내

자세한 계산 기준은 `context/pricing-design.md`를 따른다.

## 데이터 수집

### 수집 대상 사이트

데이터는 아래 사이트에서 수집하는 것을 기준으로 한다.

- 당근마켓
- 중고나라
- 번개장터
- KREAM

Open API는 사용하지 않아도 된다.
크롤링 중심으로 데이터를 수집한다.

### 수집 카테고리

우선 카테고리는 의류와 생필품 등으로 둔다.
구현 단계에서는 대표 상품을 정해서 시작한다.

### 수집 항목

수집할 데이터 항목은 아래를 기준으로 한다.

- 사이트명
- 물품명
- 카테고리
- 가격
- 게시일 또는 거래일
- 상품 상태
- 지역
- 설명
- 이미지 주소
- 원문 링크

개인정보는 수집하지 않는다.
판매자 실명, 전화번호, 채팅 내용, 주소 등은 수집 대상에서 제외한다.
자세한 크롤링 설계와 저장 형식은 `context/crawling-design.md`를 기준으로 한다.
샘플 데이터 형식은 `context/sample-data-format.md`를 기준으로 한다.
웹사이트는 저장된 데이터만 읽는다.
크롤링 실패 시 기존 저장 데이터를 유지한다.

## 데이터 가공

수집한 데이터는 웹사이트에서 쓰기 전에 정리한다.

정리 기준은 아래와 같다.

- 같은 물품 또는 비슷한 물품끼리 묶는다.
- 가격이 비정상적으로 낮거나 높은 데이터는 확인한다.
- 상품 상태와 구성품 차이를 가격 판단 기준에 반영한다.
- 날짜 기준으로 최근 데이터가 보이게 정리한다.

이상치는 전체 흐름과 크게 다른 값을 뜻한다.
예를 들어 0원, 99999999원, 부품만 판매하는 글이 포함될 수 있다.
시세 그래프와 가격 계산 기준은 `context/pricing-design.md`를 따른다.

## 경매 흐름

1. 판매자가 물품 등록 화면에 들어간다.
2. 판매자가 물품명과 상품 정보를 입력한다.
3. 웹사이트가 비슷한 물품의 시세 그래프를 보여준다.
4. 판매자가 시세를 참고해 시작가와 마감 시간을 정한다.
5. 물품이 경매 목록에 등록된다.
6. 구매자가 경매 물품을 검색한다.
7. 구매자가 경매 상세 화면에서 시세 그래프와 현재 입찰가를 확인한다.
8. 구매자가 입찰한다.
9. 마감 시간이 지나면 가장 높은 입찰자가 낙찰된다.
10. 판매자가 물품을 관리자에게 보낸다.
11. 관리자가 물품을 받고 상태를 검수한다.
12. 검수를 통과하면 관리자가 구매자에게 물품을 보낸다.
13. 판매자와 구매자가 거래 상태를 확인한다.

## 기술 기준

- 중심 기술: Python, HTML
- 웹 프레임워크: Flask
- 데이터 저장: SQLite
- 샘플 데이터 작성: CSV
- 웹사이트 조회 기준: data/app.db
- 원본 데이터 보관: JSON 선택 사용
- 기본 크롤링 방식: requests + BeautifulSoup
- 크롤링 보조 라이브러리: BeautifulSoup
- 동적 페이지 보조 도구: Selenium
- 웹사이트 데이터 사용 방식: 저장된 SQLite 데이터만 사용
- CSV 사용 방식: 샘플 작성 후 SQLite에 적재
- 그래프 표시: Chart.js
- 데이터 수집: Python 기반 크롤링
- 화면 구현: HTML 중심

## 성공 기준

- 사용자가 경매 물품을 등록할 수 있다.
- 사용자가 경매 물품에 입찰할 수 있다.
- 경매 마감 뒤 낙찰 결과를 확인할 수 있다.
- 낙찰 뒤 관리자 검수 상태를 확인할 수 있다.
- 판매 등록 화면에서 시세 그래프를 확인할 수 있다.
- 경매 상세 화면에서 시세 그래프를 확인할 수 있다.
- 가격 데이터가 시작가와 입찰 판단에 연결되어 보인다.
- 발표에서 경매 웹사이트가 주 목적임이 명확하게 보인다.

## 확정된 범위 결정

- 회원 기능은 제외한다.
- 결제 기능은 제외한다.
- 낙찰 뒤 거래는 관리자 검수 중개 방식으로 표현한다.
- 실제 배송사 연동은 제외하고, 화면 상태로만 흐름을 보여준다.
- 기술 스택은 Flask, SQLite, requests, BeautifulSoup, Chart.js로 진행한다.
- 샘플 데이터는 CSV로 작성한다.
- 대표 상품 1개당 샘플 데이터는 5개만 준비한다.
- 웹사이트는 SQLite를 기준으로 조회한다.
- 크롤링 원본은 필요할 때만 JSON으로 보관한다.
- 크롤링은 requests + BeautifulSoup을 기본으로 한다.
- 필요한 경우 Selenium을 보조로 사용할 수 있다.
- 웹사이트는 실시간 크롤링을 하지 않고 저장된 데이터만 사용한다.
- 크롤링 실패 시 기존 저장 데이터를 유지한다.
- 참고 시세는 비슷한 조건 상품의 중간 가격을 기준으로 계산한다.
- 시세 범위는 낮은 쪽 25% 가격부터 높은 쪽 25% 가격까지로 표시한다.

## 확인 필요 항목

### 1. 경매 규칙

1. 권장안: 고정 마감 시간, 최소 입찰 단위, 최고가 낙찰
   - 실제 경매와 비슷하고 구현도 가능하다.
   - 발표에서 설명하기 쉽다.
2. 간단안: 마감 시간 없이 최고 입찰가만 갱신
   - 구현이 쉽다.
   - 경매 느낌이 약해질 수 있다.
3. 확장안: 마감 직전 입찰 시 자동 연장
   - 실제 경매 서비스에 더 가깝다.
   - 시간 처리 로직이 복잡해진다.

### 2. 수집할 대표 상품

1. 권장안: 의류 2개, 생필품 2개로 시작
   - 발표용 데이터를 만들기 쉽다.
   - 카테고리 비교도 가능하다.
2. 간단안: 의류 상품만 먼저 수집
   - 데이터 기준을 맞추기 쉽다.
   - 서비스 범위가 좁아 보일 수 있다.
3. 확장안: 의류, 생필품, 전자기기까지 수집
   - 서비스가 풍부해 보인다.
   - 크롤링과 데이터 정리 부담이 커진다.

### 3. 관리자 검수 상태 범위

1. 권장안: 판매자 발송, 관리자 수령, 검수 결과, 구매자 발송 상태만 표시
   - 과제 범위에 맞다.
   - KREAM과 유사한 흐름을 설명하기 쉽다.
2. 간단안: 검수 통과 여부만 표시
   - 구현이 쉽다.
   - 관리자 중개 흐름이 약하게 보일 수 있다.
3. 확장안: 검수 사유와 사진 업로드까지 표시
   - 서비스가 더 구체적으로 보인다.
   - 화면과 데이터 입력 항목이 늘어난다.

### 4. 사이트별 크롤링 정책 최종 확인

1. 권장안: 발표에는 정책 확인 필요를 명시하고, 구현은 테스트 데이터 또는 허용 범위 데이터로 진행
   - 안전하다.
   - 발표에서 리스크 관리가 보인다.
2. 간단안: 실제 크롤링 대신 샘플 데이터를 사용
   - 구현 부담이 낮다.
   - 데이터 수집 프로젝트라는 특징이 약해질 수 있다.
3. 확장안: 사이트별 정책을 모두 확인하고 수집 범위를 문서화
   - 가장 명확하다.
   - 조사 시간이 필요하다.

### 5. 크롤링 주기

1. 권장안: 발표용으로 수동 실행 후 저장
   - 팀 프로젝트 범위에 맞다.
   - 데이터 수집 결과를 안정적으로 보여줄 수 있다.
2. 간단안: 미리 만든 CSV 데이터를 SQLite에 넣어서 사용
   - 구현이 쉽다.
   - 크롤링 결과라는 설명이 약해질 수 있다.
3. 확장안: 수동 실행 스크립트로 여러 사이트를 한 번에 갱신
   - 수집 결과를 갱신하기 편하다.
   - 오류 처리와 로그 관리가 필요하다.

## 최신 반영 사항

- 사이트별 크롤링 정책 확인 결과는 `context/crawling-policy-check.md`에 정리한다.
- 노트북 발표 시연 흐름은 `context/demo-guide.md`를 따른다.
- 웹사이트는 실제 사이트를 실시간으로 크롤링하지 않는다.
- 시연 화면은 저장된 SQLite 데이터를 사용한다.
- Chart.js는 로컬 정적 파일로 포함해 인터넷 연결이 없어도 그래프가 뜨게 한다.
